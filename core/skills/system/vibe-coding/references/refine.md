# Stage Reference: refine

Refine `design-docs/requirements.md` for clarity, grammar, and logic without changing the requested scope.

## When to Use

- The user wants to clean up the requirement before planning
- The current `requirements.md` is rough or inconsistent
- The requirement may reference an existing repo, website, game, app, clone, port, remake, or reimplementation

## Steps

### Step 1: Read the Requirement

Read `design-docs/requirements.md` and identify wording, grammar, and logic issues.

### Step 2: Check for Reference-Project Work

If the requirement asks to learn from, clone, port, remake, reimplement, or get inspiration from a source-code repo, live website, game, or app, follow `references/refine-reference-project.md`.

Otherwise continue with the normal refinement steps below.

### Step 3: Refine Only

Fix English, clarity, consistency, and obvious logic issues. Do not add implementation plans or new product scope unless the user explicitly asks.

### Step 4: Save the Refined Requirement

Write the improved content back to `design-docs/requirements.md` (it stays a living doc — no archive copy needed).

### Step 5: Summarize

Briefly report what was refined.
